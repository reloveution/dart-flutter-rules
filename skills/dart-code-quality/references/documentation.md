# Documentation

## Philosophy

- Explain why, not what
- Document for the user; answer where they first look
- No useless docs that restate the obvious
- Consistent terminology

## Style

- Use `///` for doc comments
- First sentence: concise user-centric summary
- Blank line after summary
- Avoid redundancy with code context
- Document getter OR setter, not both
- Be brief; avoid jargon
- Use backticks for code; specify language
- Doc comments before annotations

## What to Document

- Public APIs (priority)
- Private APIs when helpful
- Library-level overview
- Include code samples
- Parameters, return values, exceptions
- @deprecated: provide message
- Diagnosticable: describe all properties

## API Design

- Consider the user
- Documentation is part of good API design
